# #18 设计稿：增量扫描（无 watcher）

- Issue: `#18 [MVP2][P2] 增量扫描（无 watcher）`
- 日期: `2026-04-02`
- 状态: `Draft / 已完成设计确认，待实现计划`

## 1. 背景

#15 已经把全量扫描流程产品化：有进度、可取消、可观察的错误摘要。但对于日常使用来说，用户更需要的是：在资料库已经建立后，只同步新增 / 删除 / 修改过的文件，而不是每次都全量重扫。

本 issue 已经明确重新聚焦为：

- **增量扫描**
- **不包含 watcher**（watcher 独立留给 #28）

这意味着本期目标不是“实时感知磁盘变化”，而是把用户手动触发和启动时触发的扫描，从“总是全量”升级为“默认增量、必要时可全量重建”。

---

## 2. 本期目标

### 2.1 In scope

- 在已建立的资料库基础上，仅同步：
  - 新增文件
  - 删除文件
  - 修改过的文件
- 当库已存在时：
  - Settings 中的 **Rescan** 默认走增量扫描
  - 启动时 `autoScan = true` 默认走增量扫描
- 当库为空时：
  - 自动回退到全量扫描
- 缺失文件不物理删除，而是**标记失效**并在 UI 中灰态呈现
- 对暂时不可访问的根路径（权限、网络盘、挂载问题）只记错误，不批量误判失效
- 保留一个明确的**手动全量扫描 / 全量重建**入口

### 2.2 Out of scope

- 文件系统 watcher
- 实时自动去抖/防抖调度器
- hash 级内容指纹比对
- 云端/跨设备同步
- 因扫描发现文件缺失而强制中断当前播放

---

## 3. 用户已确认的产品决策

### 3.1 默认触发方式

当库已建立后：

- Settings 的 **Rescan** 默认走 **增量扫描**
- 启动时 `autoScan = true` 也走 **增量扫描**
- **仅在库为空时** 才回退到 **全量扫描**

### 3.2 文件变更的最小判定口径

MVP2 采用：

- **`file_path + mtime + size`**

不做 hash，不只看 mtime。

### 3.3 删除文件时的语义

当已入库文件在磁盘上消失时：

- **先标记失效**
- **不立即物理删除**

### 3.4 缺失 track 的 UI 呈现

缺失文件对应的 track：

- 仍然显示在普通列表里
- 以 **灰态 + 文件缺失标识** 呈现
- 主操作（播放 / 打开详情）**禁用**

### 3.5 不可访问根路径的处理

如果某个库根路径这次因为：

- 权限不足
- 网络盘离线
- 挂载点缺失

而不可访问，则：

- **不批量标记其下已有 track 为失效**
- 只记录到错误摘要
- 其它可访问路径继续扫描

### 3.6 全量扫描逃生口

虽然默认优先走增量，但仍保留：

- 一个明确的 **全量扫描 / 全量重建** 手动入口

---

## 4. 设计总览

## 4.1 用户视角的数据流

```text
库已存在
  -> 用户点击 Rescan / 启动 autoScan
  -> 系统判断：优先走增量扫描
  -> 对每个可访问根目录：
       - 发现新增文件 -> 入库
       - 发现已修改文件 -> 重解析并更新
       - 发现旧记录这次没扫到 -> 标记 missing
  -> 对不可访问根目录：只记错误，不批量改状态
  -> UI 展示“正在增量同步” + 完成摘要
  -> 缺失 track 灰态保留在列表中，主操作禁用
```

## 4.2 核心原则

- **默认增量，必要时全量。**
- **删除文件先软失效，不硬删除。**
- **不可访问路径不等于文件已删除。**
- **增量扫描是现有扫描系统的扩展，不新起一套独立状态机。**
- **本期通过“单次扫描任务 + 串行执行”满足最小批处理需求，不引入 watcher 级复杂度。**

---

## 5. 后端设计

## 5.1 数据模型扩展

建议直接在现有 `tracks` 表上补充增量扫描状态字段，而不是引入额外影子快照表。

新增字段建议：

- `file_mtime_ms INTEGER`
  - 上次成功扫描到该文件时的 mtime
- `file_size_bytes INTEGER`
  - 上次成功扫描到该文件时的 size
- `availability TEXT NOT NULL DEFAULT 'available'`
  - 取值：`available | missing`
- `last_seen_scan_at INTEGER`
  - 最近一次在“可访问根目录”中确认存在的时间戳

### 为什么不引入快照表

虽然单独的扫描快照表在理论上边界更清晰，但对 MVP2 来说成本过高：

- schema 更复杂
- query / join 逻辑更多
- 当前需求只需要最小闭环，不需要独立审计历史

因此本期采用：

- **直接在 track 记录上保存增量判定所需状态**

---

## 5.2 增量扫描算法

对每次增量扫描，输入仍然是一组 library roots。

### Step A：先处理根路径可访问性

对每个 root：

- 若 root **可访问**：进入增量同步流程
- 若 root **不可访问**：
  - 记录错误摘要
  - 跳过该 root
  - **不对其下已有 track 执行 missing 标记**

### Step B：遍历当前磁盘文件

对 root 下当前能看到的媒体文件逐个处理。

#### 情况 1：数据库中不存在该 `file_path`
- 判定为 **新增文件**
- 正常走 metadata 解析 + 入库
- `availability = available`
- 写入新的 `file_mtime_ms` / `file_size_bytes`
- `last_seen_scan_at = now`

#### 情况 2：数据库中存在该 `file_path`，且 `mtime + size` 未变化
- 判定为 **未变化**
- 跳过重新解析
- 仅更新：
  - `availability = available`
  - `last_seen_scan_at = now`

#### 情况 3：数据库中存在该 `file_path`，但 `mtime` 或 `size` 变化
- 判定为 **已修改**
- 重新解析 metadata
- 更新 track 记录
- `availability = available`
- 更新 `file_mtime_ms` / `file_size_bytes`
- 更新 `last_seen_scan_at = now`

> 这里的“重新解析”也应自然覆盖 #19 这类伴生资产（例如封面、歌词等依赖文件结果）。

### Step C：标记缺失文件

当一个 root 扫描完成后，再对该 **可访问 root** 执行“缺失标记”：

- 找出数据库中归属于该 root 的已有 track
- 若该 track 这次没有在磁盘遍历结果中出现
- 则标记：
  - `availability = missing`

注意：

- **不物理删除记录**
- 只做 soft-invalid / soft-missing

### Step D：missing 文件恢复

如果一个之前 `missing` 的文件后来又重新出现：

- 自动恢复为 `availability = available`
- 按当前磁盘文件重新解析并更新 metadata

---

## 5.3 与现有扫描系统的集成

当前项目已经有：

- 扫描任务状态
- 进度上报
- 取消能力
- 错误摘要

本期不应重新发明一套扫描状态系统，而应扩展现有 contract。

建议在现有 `ScanStatus` / `ScanSummary` 基础上增加：

- `scan_mode = 'full' | 'incremental'`
- `updated_tracks`
- `missing_tracks`
- `restored_tracks`（可选，但推荐）

这样前端可以准确表达：

- 这次是不是增量同步
- 这次实际做了哪些同步动作

---

## 5.4 启动与手动触发语义

### 启动 autoScan
- 若 `autoScan = true` 且库为空：走全量扫描
- 若 `autoScan = true` 且库非空：走增量扫描

### Settings → Rescan
- 默认走增量扫描

### Settings → Full Rebuild / Full Scan
- 明确手动入口
- 语义上是“重新全量遍历全部 root，按全量模式重建/同步”

---

## 5.5 去抖 / 批处理

Issue #18 提到“同一批文件变更有去抖 / 批处理能力”。在**无 watcher**前提下，本期最小实现定义为：

- 每一次用户触发 / 启动触发，本身就是一个批次
- 继续沿用当前“已有扫描在进行中时拒绝第二次启动”的串行策略
- 不额外实现复杂调度队列或定时器合并

也就是说，本期通过：

- **单任务串行执行**
- **一次触发处理整组 roots**

来满足最小批处理要求。

## 6. 前端 / UI 设计

## 6.1 Settings 中的扫描入口

UI 建议收敛为两类入口：

### 主入口：增量同步
- 文案可用：`Rescan` / `Incremental Scan`
- 当库已建立时，这就是默认日常同步入口

### 次入口：全量重建
- 文案应更明确，例如：
  - `Full Scan`
  - `Full Rebuild`
- 需要提示“耗时更长” / “会重新遍历全部库路径”

---

## 6.2 扫描状态反馈

前端应能明确区分：

- `正在增量同步…`
- `正在全量扫描…`

完成后摘要建议至少包含：

- 新增多少
- 更新多少
- 标记缺失多少
- 恢复多少（如有）
- 错误多少

这样用户才知道“这次增量到底做了什么”。

---

## 6.3 缺失 track 的列表语义

缺失 track 在 MVP2 中应保持：

- **可见**
- **灰态**
- 带一个明确标识，例如：`文件缺失`

### 主操作约束

对这类 track：

- 播放禁用
- 打开详情主操作禁用

### 最少应覆盖的 surface

建议至少在以下表面统一体现：

- Songs 列表
- Album Detail 中的 track 列表
- Playlist Detail 中的 track 列表（若复用通用 track row，最好顺带接入）

目标是避免：

- 一个地方灰了
- 另一个地方却看起来正常

---

## 6.4 不可访问 root 的错误提示

当某个 root 因权限/网络盘/挂载问题不可访问时：

- 本次扫描继续处理其它 roots
- 只在错误摘要中提示

建议用户文案类似：

- `1 个库路径暂时不可访问，已跳过；其原有曲目未标记为缺失。`

这能减少误解，尤其是外接硬盘 / 网络盘用户。

---

## 6.5 当前播放中的缺失文件

若某首歌已经开始播放，而扫描过程中发现其磁盘文件已缺失：

- **本次不强制打断当前播放**
- 但 UI 状态会在后续刷新后显示为 `missing`
- 用户下次再从列表主动触发它时，因为主操作禁用，不能再次播放

这样做的好处是：

- 不在扫描中途打断体验
- 又能保持后续语义一致

## 7. 测试策略

## 7.1 Rust / 后端

至少覆盖：

1. 新文件能被增量入库
2. `file_path + mtime + size` 变化时会重解析更新
3. 文件删除后会被标记 `missing`
4. missing 文件恢复后会回到 `available`
5. 不可访问 root 不会批量标记 missing
6. 多 root 场景下，一个 root 出错不影响其它 root
7. autoScan / Rescan 共用同一增量逻辑入口
8. 增量扫描状态摘要正确返回新增 / 更新 / 缺失 / 恢复计数

## 7.2 前端

至少覆盖：

1. Songs / Album 列表里 missing track 灰态显示
2. missing track 主操作禁用
3. Settings 中能区分增量同步与全量扫描入口
4. 扫描状态能显示“正在增量同步”
5. 完成摘要能看到新增 / 更新 / 缺失计数

## 7.3 回归验证

实现完成后，最少需要：

- `just qa`
- #18 对应的 Rust 增量扫描回归测试
- 前端缺失状态 / 扫描状态回归测试
- 一次手工烟测：
  - 库已建立后，Rescan 默认走增量
  - 删除一个文件后，track 变灰并禁用播放
  - 恢复文件后，track 恢复正常
  - 一个 root 离线时，错误摘要可见，但不批量误判 missing
  - Full Scan 入口仍可用

## 8. 风险与边界

### 风险 1：软失效会让查询逻辑变复杂

应对：

- 明确哪些列表默认显示 missing，哪些交互禁用
- 尽量把 availability 判断收敛到通用 query / row model / UI row 组件层

### 风险 2：不可访问 root 与真实删除容易混淆

应对：

- 只有在 root 可访问时，才对其下旧记录执行 missing 标记
- 错误摘要必须明确“已跳过，不判失效”

### 风险 3：全量与增量双入口会让用户困惑

应对：

- 默认只强调 Rescan（增量）
- Full Scan 作为次级高级入口，文案强调“耗时更长 / 用于修复库状态”

---

## 9. 本期结论

#18 的 MVP2 设计收敛为：

- **库已建立后默认优先增量扫描**
- **判定口径为 `file_path + mtime + size`**
- **删除文件先标记 missing，不物理删除**
- **missing track 在 UI 中保留、灰态、主操作禁用**
- **不可访问 root 不批量判失效，只记错误摘要**
- **保留一个手动全量扫描 / 全量重建入口**
- **本期不做 watcher / hash / 实时调度器**
